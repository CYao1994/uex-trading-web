# PRD — UEX Trade Navigator 第三阶段

## 项目信息

- **Language**: 中文
- **Programming Language**: React + Vite + MUI + Tailwind CSS（前端），FastAPI + Python（后端）
- **Project Name**: uex_trading_web_phase3
- **原始需求**：链式跑商补充货物逻辑 + 物品数据库查询功能

---

## 产品目标

1. **最大化货仓利用率**：链式跑商每段不再只运一种商品，而是填满货仓/花完预算，提升单段利润上限
2. **提供全维度数据浏览能力**：用户可脱离路线规划场景，独立浏览和查询商品/飞船/终端的完整信息
3. **保持系统简洁性**：补充货物逻辑遵循同目的地约束，避免组合爆炸；物品数据库复用已有后端接口，零新增外部依赖

---

## 用户故事

1. **As a 跑商玩家**, I want 链式跑商每段自动填充可盈利的副商品 so that 我不会浪费货仓空间和剩余预算，单趟利润更高
2. **As a 跑商玩家**, I want 在链式结果中看到每段所有运输商品的明细 so that 我能在游戏中准确执行多货物的装卸操作
3. **As a 新手玩家**, I want 浏览所有商品和飞船的数据 so that 我不需要切到 UEX 官网就能了解物品属性
4. **As a 资深商人**, I want 按类别和星系筛选物品 so that 我能快速定位特定区域的交易机会
5. **As a 跑商玩家**, I want 从物品详情直接跳转到价格查询 so that 我的工作流不被中断

## 待确认问题

1. **补充货物的 SCU 来源冲突**：同一种补充商品在当前出发地的多个 terminal 可能都有卖，价格可能不同。是否取最低买入价？还是按 terminal 分别列出？
   - *当前假设*：取最低买入价（与主商品选择逻辑一致，同一 location 多 terminal 合并取最优）

2. **补充货物的目的地库存约束**：补充商品卖出时，目的地的 `scu_sell_stock` 是否需要与主商品共享？例如主商品占了 50 SCU 的收购额度，补充商品只能用剩余额度？
   - *当前假设*：各商品独立计算库存，不做共享扣减（简化逻辑，实际游戏中玩家也是分别卖出的）

3. **物品数据库是否需要后端新接口**：目前 `/commodities` 返回的 `CommodityOption` 只有 `id/name/name_zh`，列表展示是否需要更多信息（如商品类别）？
   - *当前假设*：列表只展示基本信息，详情通过 `/commodity-prices/{id}` 获取完整价格数据。如需类别字段，可扩展 `CommodityOption` schema 增加 `category` 字段

4. **物品数据库的终端详情展示**：终端的"所属商品"信息需要逐 terminal 查询价格数据，可能较慢。是否仅在用户主动点击时懒加载？
   - *当前假设*：终端详情只展示位置信息，不主动加载商品列表；商品列表通过价格查询 Tab 间接访问

5. **移动端布局**：物品数据库在移动端是否需要调整为上下布局（列表在上、详情在下）？
   - *当前假设*：移动端使用上下布局，桌面端使用左右布局

6. **链式跑商向后兼容**：`ChainLeg` 新增 `commodities` 字段后，旧版前端（未升级）是否能正常忽略该字段？
   - *当前假设*：可以，Pydantic 模型新增可选字段不影响旧客户端，旧前端的 `commodity_name` 等字段仍保留
