# PRD — UEX Trade Navigator 第四阶段

## 项目信息

- **Language**: 中文
- **Programming Language**: React + Vite + MUI + Tailwind CSS（前端），FastAPI + Python（后端）
- **Project Name**: uex_trading_web_phase4
- **原始需求**：响应式全面适配 + 飞书反馈功能

---

## 产品目标

1. **全设备可用性**：确保 UEX Trade Navigator 所有页面和组件在手机（≥375px）、平板（≥768px）、桌面端（≥1280px）下均可用且体验流畅，消除移动端布局错位、溢出、不可操作等问题
2. **用户反馈闭环**：通过飞书 Webhook 建立结构化反馈通道，让用户便捷提交 Bug 报告和功能建议，运营侧即时收到通知，缩短问题发现→修复的周期
3. **移动端体验不降级**：响应式适配不仅是"能用"，更要"好用"——移动端保留核心交互效率，关键操作（路线计算、价格查询）触屏友好

---

## 用户故事

1. **As a 手机端跑商玩家**, I want 在手机浏览器上正常使用路线计算和价格查询 so that 我在游戏外也能快速查路线而不需要打开电脑
2. **As a iPad 用户**, I want 所有面板在竖屏和横屏下都布局合理 so that 我不必频繁缩放和横滚页面
3. **As a 遇到 Bug 的玩家**, I want 在应用内直接提交反馈并附上截图 so that 我不需要切换到其他渠道报告问题
4. **As a 想提功能建议的玩家**, I want 提交功能建议时能选择分类 so that 我的建议能被开发团队更高效地处理
5. **As a 移动端用户**, I want 汉堡菜单导航和触屏操作流畅 so that 我能快速在不同功能 Tab 间切换

## UI 设计建议

### 响应式断点策略

```
断点定义（基于 MUI 默认断点）：
- xs: 0-599px    （手机竖屏）
- sm: 600-899px  （手机横屏/小平板）
- md: 900-1279px （平板/小笔记本）
- lg: 1280-1919px（桌面）
- xl: 1920px+   （大屏）

核心适配逻辑：
- xs/sm → 移动端布局
- md → 平板过渡布局
- lg/xl → 桌面布局（当前已有体验）
```

## 待确认问题

1. **飞书 Webhook URL 来源**：需要提供飞书群机器人的 Webhook 地址。是新建群还是复用已有群？是否需要签名验证（`sign` 字段）？
   - *当前假设*：使用飞书群自定义机器人 Webhook，暂不启用签名验证（可后续加）

2. **截图上传方案**：飞书 Webhook 消息不支持直接内嵌图片二进制，需要先上传到图片托管服务再引用 URL。可选方案：① 后端转存到腾讯云 COS（需配置）；② 前端转 base64 嵌入消息文本；③ 仅发送截图的文字描述，不传图
   - *当前假设*：前端将截图压缩后转 base64，作为消息中的文本描述附带（飞书富文本支持图片上传需额外 API，首期简化处理）

3. **移动端底部 Tab Bar vs 汉堡菜单**：底部 Tab Bar 更符合移动端操作习惯，但当前已有汉堡菜单实现。是否需要替换？还是两者并存（Tab Bar 放核心 4-5 个，汉堡菜单放全部 6 个 + 反馈入口）？
   - *当前假设*：移动端保留汉堡菜单但新增底部 Tab Bar，底部放核心 4 个 Tab（清仓/进货/链式/价格），其余入口在汉堡菜单中

4. **反馈频率限制实现**：基于 localStorage 的简单限频是否足够？是否需要后端 IP 限频？
   - *当前假设*：前端 localStorage 5 分钟限频 + 后端可选 IP 限频（首期仅前端限频）

5. **PWA 是否纳入本期范围**：PWA 支持（离线缓存、添加到主屏幕）能显著提升移动端体验，但工作量较大。是否作为 P2 延后？
   - *当前假设*：P2 延后，本期专注布局适配和反馈功能

6. **DatabasePanel 移动端详情交互**：移动端点击列表项后，详情展示采用"底部抽屉"还是"替换列表"模式？
   - *当前假设*：首期采用"替换列表"模式（详情替换列表区，顶部有返回按钮），后续可升级为底部抽屉
