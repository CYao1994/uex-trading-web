## Project context
_What is this project? What's its goal? High-level identity._

**UEX Trade Navigator** (v3.22.0) — A web-based trading route planner for Star Citizen players, targeting the Chinese community. Users input cargo/origin to get optimized sell/buy routes (shortest distance, highest profit). Deployed on EdgeOne Pages.

Tech stack: React 19 + MUI 9 + Tailwind CSS 4 (frontend), Python FastAPI (Cloud Functions backend). Backend proxies UEX API for game commodity/terminal data.

GitHub: `https://github.com/CYao1994/uex-trading-web`

## Architecture decisions
_Major design choices with rationale. The "why" matters more than the "what" for future sessions._

- **EdgeOne Pages deployment**: SPA frontend + Python Cloud Functions backend, all on one platform
- **TTL caching**: Backend uses 12 cache instances with TTLs (2h prices, 6h terminals/commodities, 24h distances). Frontend uses localStorage with endpoint-specific TTL (15min for prices, 24h for static data)
- **Tab-based routing**: Single-page app with lazy-loaded panels per feature (sell/buy/warbond/chain/ship_components/ship_weapons)
- **Dark sci-fi theme**: Amber/gold (#c9a227) primary, deep space background (#060a13), Orbitron + Rajdhani fonts
- **urllib.request over requests**: EdgeOne Cloud Functions may not have curl binary; Python stdlib urllib works reliably
- **SSL verification disabled**: EdgeOne CA bundles may be outdated; skip verification to avoid TLS errors with UEX API
- **Terminal loading per star system**: Default UEX endpoint only returns Stanton; must query each system (Stanton=68, Pyro=64, Nyx=55) separately
- **Split routing**: When single terminal can't fulfill quantity, split to max 3 terminals; avoids partial-fulfillment bottlenecks
- **Non-trading terminal filtering**: 30+ terminal prefixes excluded (food shops, locker rooms, etc.) that UEX tags as type=commodity but don't trade in-game
- **EdgeOne Pages file size limit**: Deployment platform has per-file size limits. Must avoid large single files. Use code splitting, lazy loading, and keep data files under reasonable sizes. Images especially must be compressed.
- **DDS textures not extracted**: First unp4k run only extracted DataForge XML (game.dcb). DDS texture files in `Data/Objects/planets/` need separate extraction via `unp4k.exe Data.p4k *.dds`. These are binary assets inside .pak sub-archives within Data.p4k.
- **Station/city coordinates**: NOT in DataForge XML records. Only star system galactic positions exist (Stanton/Pyro/Nyx). Individual station XYZ positions are in CryEngine binary scene files (.cry) which unforge cannot parse. Would need Starfab tool or community coordinate datasets.
- **Star map approach**: Use 3 star system coordinates + parent hierarchy from starmap XMLs + orbital radius approximation for planet positioning. Fall back to hash-based positioning for stations without precise coords. Procedural textures (shader-based) avoid large DDS file dependencies. 3D star map implemented and working.
- **Star map dependencies**: three, @react-three/fiber, @react-three/drei installed via npmmirror. Panel: `StarMapPanel.jsx` + `starmap/StarMapScene.jsx`. 2D list fallback for mobile. Tab key: `starmap`.
